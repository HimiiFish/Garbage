# 第1章 绪论

## 1.1 选题背景
这几年科技圈最热闹的新闻，莫过于几家商业火箭公司隔三差五的发射任务。大家都在抬头看卫星上天，但我更关注那些留在天上掉不下来的废铁。近地轨道的环境其实早就快失控了。根据所谓的“凯斯勒现象”（Kessler syndrome），轨道碎片如果密集到一定程度，互相撞击会产生更多碎片，最终引发失控的链式反应。想想看，在距地面 400 公里的轨道上，物体的飞行速度大概是 7.8 km/s，一小片剥落的油漆砸上去，动能比一整箱炸药还要可怖。

大学四年的软件工程课，我绝大部分时间都在跟增删改查、后台管理系统、死板的表单打交道。写久了真的会怀疑自己只会当个搬砖工。我想要画面，想要能够动起来的东西，想用代码弄出点带物理法则的乐子。如果把清理太空垃圾这个冷门话题做成一款电子游戏呢？让玩家去算燃料，去抓着机械爪碰运气。带着这个模糊的想法，我把毕业设计方向定在了一款 2D 视角的太空垃圾清理模拟游戏上。

## 1.2 研究目的与意义
很多人毕设做的游戏，勉强点两下不报错就能上台答辩了，稍微玩久一点内存就会爆掉。我不想弄个一碰就碎的纸糊外壳。我要写出一个地基扎实、跑上个把小时帧率也不会崩塌的程序。

我以前用 Unity 写物理反馈，贪图方便，经常让两个脚本互相硬生生地调来调去。子弹撞到人，就在子弹的脚本里去拽 UI 扣血条；哪天我把血条节点删了，游戏当场死机。这次我逼自己戒掉这毛病。我引入了 UniRx 库，用消息总线把底层的物理运算和表层的动画表现切断联系。那些在《软件架构》课上听得打瞌睡的设计模式，这次我必须在代码里把它们用活。

最大的阻碍倒不在代码怎么写，而在怎么把枯燥的物理算式变得有娱乐性。极坐标换算、引力拉扯、动量守恒，这些东西躺在纸上很严谨，但照搬进游戏里简直是灾难。如果连变个轨都要玩家在脑子里解微积分，大家五秒钟就会删游戏。我花了几个星期的时间去改那些碰撞体积和推力系数参数。在真实受力和游戏手感之间找到那个能自我欺骗的平衡点，几乎耗尽了我的耐心。

## 1.3 国内外研究现状
刚开始的时候，我去翻了不少技术社区的名篇，想搞清楚做这类带有物理拟真元素的游戏，前人都踩了什么雷。

### 1.3.1 国外探讨与拟真妥协
圈子里最常拿来做标杆的就是《星际拓荒》（Outer Wilds）还有《坎巴拉太空计划》（KSP）。这两款游戏处理物理的方式截然不同。KSP 用的是拼接圆锥曲线近似（Patched Conics Approximation），它不去算复杂的 N 体问题，而是假定飞船在任意时刻只受一个大天体的引力影响。这种妥协换来的是玩家能够在屏幕上直观地看到轨道的预判线。<br/>很多海外开发者在维基和 GDC 演讲上反复强调一个教训：别拿学术界的精确度去要求游戏引擎。浮点数误差会让完美的物理公式在运行几万帧后崩溃。所以，核心机制必须去适配玩家的情绪曲线。如果引擎算出来的轨道和玩家心里以为的轨道不一致，错的永远是引擎，你得用代码去修正它。

### 1.3.2 国内工程视角的演进
视线收回到国内的技术论坛，风气变得特别务实。我上大二那会，到处都是教你怎么写酷炫的着色器（Shader）；现在大家都在聊发热、聊卡顿、聊内存泄漏。国内好几个独立爆款（比如《戴森球计划》）的团队出来分享经验时，满篇讲的都是数据导向设计（ECS）和对象池（Object Pool）。拿引擎自带的功能去跑大阵列的物体，C# 会被垃圾回收（GC）产生的瞬间卡顿彻底拖死。这就给我提了个醒——这毕设不能光图跑通就完事了，既然屏幕上要刷那么多太空垃圾碎片，我就得从一开始管好每个对象的生老病死。

## 1.4 研究内容与技术路线
工程推进没有我想的那么浪漫，我把它拆成了四步。

第一步，砍需求。不写设计文档直接敲键盘的下场就是无限返工。我定死了飞船的操作极限、抓勾的投射距离，明确告诉自己：不搞三维，不搞流体模拟。

第二步，搭架构。配置完 Unity，我先把底层通讯的线牵好。发布/订阅机制搭起来的时候有点折磨，但一旦逻辑层能通过总线盲发信号，后面就算 UI 面板代码全出了问题，游戏的物理进程依然能闭着眼睛跑。

第三步，死磕数学。这部分卡了我最久。在 2D 平面里让飞船绕着原点做圆周运动，用直角坐标（XY）算出来的结果总有诡异的误差。最后我全面倒向了极坐标系，靠着追踪角度 $\theta$ 和 半径 $r$。这涉及到把坐标转来转去，但至少不用担心飞船在转弯的时候把速度给弄丢了。

第四步，拷打测试。我把生成的同屏垃圾数量锁死在 80 个，主要是我的破笔记本跑到两三百个就开始掉帧。调整引力系数是个体力活，我加了一个人工黑洞，测试时发现飞船靠近黑洞中心时受力会趋近于无限大，当场被弹射出整个场景边界。我不得不去给分母强行加上一个下限。

## 1.5 论文组织结构
第一章是绪论。铺垫一下物理背景，以及目前的开发风气。
第二章聊技术栈。主要解释为什么 C# 原生事件满足不了我，得靠外挂的 UniRx 续命。
第三章理清需求。玩家到底按什么键，游戏不能容忍什么 Bug。
第四章解析系统架构。记录怎么把画面、逻辑和数据给硬生生撕开。
第五章是最麻烦的代码拆解。把变轨的三角函数和抓勾的协程状态机拉出来鞭尸。
第六章是测试记录。展示那些因为浮点数溢出或者碰撞穿模引发的惨案。
第七章随便写写总结。骂一骂代码里的烂摊子，想想以后怎么用对象池收拾残局。

# 第2章 相关技术概述

## 2.1 C#/.NET 与响应式扩展（UniRx）
整个后端的骨架是用 C# 撑起来的。泛型和委托机制很棒，但我大二曾经被原生的 C# `event` 坑惨过。原生的事件机制有个致命缺陷：如果一个游戏物体被引擎销毁了，但它死前忘了在中央管理器里注销（Unsubscribe）自己的回调函数，这块内存将永远挂在那里。C# 的垃圾回收器看着这种长命引用的“死物”毫无办法，玩半小时游戏内存能吃掉好几个 G。

为了彻底摆脱这种隐性内存泄漏，我用 UniRx 替换了原生事件。Rx 系列的奥义就是把所有的事件当成随时间流动的“流”。配合 `TakeUntilDestroy(this)` 这个神奇的后缀，订阅关系直接和脚本的生命周期绑定。飞船一旦因为碰撞被清除，它身上挂靠的所有事件监听当场自动熔断销毁，走得干干净净。这让我从繁琐的析构清理中解脱了出来。

## 2.2 Unity基于组件的物理系统
说到 Box2D 物理模块，我在这上面吃了不少暗亏。起初我弄了一些高速飞行的太空垃圾碎片，满心欢喜地等着抓勾去拦截它们，结果在测试时，那些碎片竟然眼睁睁地穿过了抓勾，毫发无损。这是由于普通的离散碰撞检测（Discrete）导致的。在第 1 帧时，碎片在抓勾前面；第 2 帧计算时，碎片速度太快，直接越过了抓勾的厚度跑到后面了。物理引擎一看，两帧都没引发重叠，自然判定无事发生。我被迫只能把抓勾和高速碎片的刚体全改成了连续碰撞检测（Continuous Collision Detection, CCD）。这虽然对 CPU 不太友好，但总算解决了“量子穿隧”的笑话。

我还发现把摄像机的运镜逻辑直接扔进 `Update` 绝对是个灾难。画面会伴随帧率的抖动出现令人头晕的撕裂感。我切到了 Cinemachine 虚拟相机，花了几晚猛调参数，给相机的追踪套上了一个略微迟缓的弹性阻尼，这画面才总算像是在太空里漂游了。

## 2.3 消息与事件驱动架构 (EDA)
我非常恨写轮询。如果每一帧游戏都有个脚本在问“玩家现在油跑光了吗？”，那等于让 CPU 疯狂空转。事件驱动架构（EDA）就轻松多了，没事情发生大家就都闭嘴。

我用结构体打包了游戏的所有关键状态。比方说 `class FuelEmptyEvent {}`。控制油箱的脚本只管看着扣油条，到了 0 就往外头抛这个事件。音乐管理器收到事件马上切悲情 BGM，主摄像机收到事件立马拉远焦距，死亡面板收到事件就弹出来。这几个干活的脚本互相甚至不知道对方的存在。我不必在油箱脚本里塞进去成吨的各个 UI 的引用路径，这在逻辑上干净得让人身心舒畅。

## 2.4 基于Resources后备加载的文本调度技术
如果把对话或者科普台词硬写进业务代码里（硬编码），那我每次改个错字都得让 Unity 把这几千个文件重新编译一次。我也想过用当前很火的 Addressables 系统去管理资源，但搞这么个企业级的玩意只为了读两行 TXT 文本，实在大可不必。

我挑了最旧但也最实在的 `Resources.Load` 接口。启动游戏的时候，后台顺着路径去读本地文件，按照规定的停顿符用 `Split` 砍成几百句台词，全部压进队列进程里备着。要展示一句就出列一句。简单粗暴，而且就算是随便拿个记事本程序也能随时给游戏换对白。

# 第3章 游戏需求分析

## 3.1 总体目标与用户要求
一开始我想还原一套硬核引力机制，但在我室友试玩并疯狂坠毁了三次之后，我决定放过玩家。它应该是一个披着太空外衣的“抓娃娃机”。必须让玩家在不理解什么是“比冲”、什么是“逃逸速度”的情况下，也能直观地感受到重力和燃料带来的焦虑。

## 3.2 核心功能需求

### 3.2.1 轨道控制与碰撞判定
在这个游戏里，没有寻常的双摇杆自由飞行，你只能像卫星一样绕圈。在真实的轨道力学里，加速（Prograde）并不会让你单纯变快，而是会抬高你轨道另一侧的高度（远地点）。同理，减速会让近地点掉近大气层。这叫做霍曼转移（Hohmann transfer orbit），但这玩意写进俯视视角的小游戏里，操作反直觉到令人发指。为了保住游戏性，我用代码作弊了：玩家按向上键，游戏就在后台径直加上你的轨道半径，强行修正你的切线速度，让你以最平滑的方式直接滑向外部的一个标准同心圆。

### 3.2.2 机械抓勾机制
飞船唯一能够还击或者收集物资的方式就是发射抓勾。在早期版本中我用了盒子碰撞体（BoxCollider），结果抓勾在伸缩的过程里经常莫名其妙卡住飞船。后来我索性把抓勾前端的真实重力检测全卸了，改写成基于物理射线（Raycast2D）的雷达探针。飞船一边向外伸探头，一边用射线沿途扫。只要那条射线刮到名为“垃圾”的层级通道，代码立刻叫停射线，直接在目标和抓勾之间通过逻辑绑定父子层级，把它死死铐回货舱。

### 3.2.3 资源限制与轮回结算
说到资源，我原本还想参考齐奥尔科夫斯基火箭方程 $\Delta v = v_e \ln\frac{m_0}{m_f}$ 来动态计算飞船质量变化对燃料的消耗。但后来算算，这真没必要。玩家只关心进度条。我最后把它简化成了纯粹的点数消耗。开方向引擎每秒扣 10 个点数，如果不幸在半道上点数见底了，系统立刻没收键盘输入权，宣布变成太空里的垃圾。你只能看着飞船循着残余速度无奈地转圈，这股绝望感远比弹个提示框给劲得多。

### 3.2.4 干扰天体与物理异常
如果只是变轨抓东西，玩到第五关就会无聊透顶，所以我丢了个能产生乱序拉扯的黑洞进去。万有引力的基础算法 $F = G \frac{m_1 m_2}{r^2}$ 里有个致命的坑：这个分母 $r^2$ 是两点之间的距离的平方。如果飞船的中心轴和黑洞由于意外完全重合，$r$ 逼近于 0，算出来的拉力 $F$ 就是天文数字。当时我亲眼看着飞船瞬间被加速到了超过物理引擎的算力极限，坐标参数直接报出 `NaN`。我不得不硬给分母卡了一个死下限 `Mathf.Clamp(distance, 2f, 100f)`，才把这横冲直撞的物理 Bug 给按住。

### 3.3 非功能需求

### 3.3.1 性能与垃圾回收（GC）控制
太空中经常有各种烟雾和爆炸效果释放。频繁调用 `Instantiate` 生成预制体，用完就 `Destroy`，这在 C# 里是大忌。它会导致内存堆极快被充满，随后引发触发垃圾回收的 CPU 停顿机制。对于一个要在低配电脑上跑到 60 帧的项目，这种停顿肉眼可见。我要求核心循环内必须禁掉那些即插即抛的内存调配。

# 第4章 系统设计

## 4.1 总体架构设计
刚写的时候没管什么架构，不到五天，我的 `ShipController.cs` 文件就膨胀到了 800 行。这就跟一个厨房里只请了一个大厨，他又要切菜又要洗碗还要去门口收银；不出岔子就有鬼了。

被逼无奈，我把代码劈成了三段，类似于 MVC 的变体。表皮层只管播动画、换粒子特效、推放音乐。逻辑核心层接管算数和受力；不管外面炸得多好看，在逻辑层的眼里这就只是一串 Vector2 加上质量的数组而已。数据仓库层全用 ScriptableObject 打包。每次调个黑洞的吸力或者刷怪频率，只需要在可视化面板里改，再也不用去茫茫代码海里找那两个常量。

## 4.2 场景状态流转
这玩意底层是个状态机。我手写了一套基本的状态调度。发车前是**待命闲置状态**，游戏开始切入**主物理循环状态**。当你没油了，代码强制中止物理刷新，切入**死亡结算状态**。如果安全凑满了碎片，玩家小心翼翼地对接上空间站，就进**靠港补给状态**。靠港后一切回归初始量，顺便从文本队列里弹两段剧情，歇口一气再重来。这几个状态之间互相上着互斥锁，死活不让它们并行。

## 4.3 核心模块详细设计

### 4.3.1 极坐标向心物理模型
我之前已经受够了在直角坐标系里算弧度。稍微有点偏差，飞船就在绕圈的时候越飘越远，没法维持个圆。所以我舍弃了 Unity 内建力系统，手动去转译位置。
空间站占据 $(0,0)$。飞船只关心当前自己离原点的直线距离 $r$ 还有倾角 $\theta$。
每过一固定极短时间 $dt$，我给 $\theta$ 叠加上一个基础角速度。如果玩家触发变轨操作，目标轨道设定为 $r_{target}$，我利用平滑插值进行过渡：
$$ r_{current} = r_{current} + (r_{target} - r_{current}) \times \text{lerpFactor} \times dt $$
算完之后再交还给 Unity 的渲染和物理检测引擎去描绘实际位置：
$$ x = r_{current} \cos\theta $$
$$ y = r_{current} \sin\theta $$
这种写法非常独裁。即使飞船遭到了一块几吨重的陨石撞击，如果我的代码拒绝更新 $r$ 的值，那飞船连晃都不会晃一下。这种极度“固执”的代码彻底杜绝了飞船偏离主航道的可能。

### 4.3.2 抓勾的状态机与射线检测
机械爪逻辑不那么费脑子，拆开了看就四步：
1. **射出去**：用鼠标点了个屏幕坐标，算一个方向出来，爪子顺着那个方向匀速顶出去。
2. **扫描**：在这个射击半途中开着物理射线的层级遮罩。撞到别的东西直接无视，必须是指定层级里的碰撞体才算数。
3. **抓取回调**：碰到了，关闭目标的动力，变成机械爪的死皮赖脸的附件，同时翻转速度向量，扯回老家。
4. **结算清退**：一碰到母船，就丢进报废池里头，广播加分。

# 第5章 关键功能的代码实现

## 5.1 事件层（MessageBroker）构建
看代码是怎么互相抛东西的。在飞船安全卸货那一瞬间，它没有去写乱七八糟的寻址代码，就只有一句干脆的广播：
```csharp
MessageBroker.Default.Publish(new GarbageCollectedEvent());
```
就这一行。它也不管谁听见了没。
然后在几十行开外的记分板脚本 `ScoreUI.cs` 的启动挂载里，挂着这么个钩子：
```csharp
MessageBroker.Default.Receive<GarbageCollectedEvent>()
    .Subscribe(_ => UpdateScoreUI())
    .AddTo(this);
```
只要飞船广播一出去，这边的记录函数就会被踹醒。整个解耦过程让我后期加特效、删音效的时候一点顾虑都没有。即便我手滑把画布整个删除了，主循环甚至连个红色的抛错都不会给出来。

## 5.2 变轨系统与防穿模的实现
所有跟位移扯上关系的大型推力，一旦丢出 `FixedUpdate()` 就会变成噩梦。普通显卡跑游戏的帧率是一直在飘的，你用忽大忽小的间隔时间去算极坐标，这会导致极高频率的轨道颤抖。用物理固定步长能掐得死死的。
穿模是另一件折腾人的事。我花了一整天排查为何拖着垃圾回母舰时经常当场自爆。原因在这：垃圾碎片的碰撞判定体被拽回来的一瞬间，蹭到了飞船自己挂在船头的死亡感应器。我在触发入口里紧急补了一行屏蔽门：
```csharp
if (other.CompareTag("CapturedTrash")) return;
```
如果打上了捕获标记，代码直接在触发死亡前刹车返回，完美绕开判定这出惨剧。

## 5.3 抓勾控制的协程流
让一个物体花个几秒钟飞到屏幕左边再回来。这种横跨几十帧的行为逻辑不能写死在主死循环里。Unity 的迭代器（Coroutine）帮了大忙。
在一个名为 `ShootClaw` 的 IEnumerator 进程中，我丢进去一个死循环：
```csharp
while(distance > 0.1f) {
    transform.position = Vector3.MoveTowards(transform.position, mousePos, speed * Time.deltaTime);
    if(CheckRaycastHit()) break;
    yield return null;
}
```
`yield return null` 这行指令让它每一帧走到这里就停工，把线程还给引擎，下一帧再从这儿断点接续干活。如果雷达响了，那个 `break` 会一脚踢翻循环，无缝掉进后面的收缩路线中去。没有各种标记变量在外面飘，代码收得很紧。

## 5.4 黑洞引力计算
当探测器捕捉到不明物体进入黑洞的地界时，`OnTriggerStay2D` 就接管了苦力活。
我手写出来的吸引算法并不复杂，核心还是计算两个坐标系的差值：
```csharp
Vector2 dir = (blackholePos - shipPos).normalized;
float dist = Mathf.Max(Vector2.Distance(blackholePos, shipPos), 1.5f);
float forceAmount = (G_CONSTANT * blackHoleMass * shipMass) / (dist * dist);
shipRigidbody.AddForce(dir * forceAmount);
```
那个至关重要的 `Mathf.Max(..., 1.5f)` 就是被无穷大推力制裁后总结出来的血泪参数。强制垫了一层下限后，再怎么贴近中心也不会把物理引擎拽崩溃了。引力这东西，配合着向心变轨的数学约束，玩家只能被迫去算计逃跑的时机。

# 第6章 项目测试与实验结果

运行前信心满满，运行后到处补锅。这里总结点实地测试时的荒唐事。

## 6.1 运行性能观测
我非常清楚自己的垃圾回收系统没做好缓存池，所以跑 Profiler 的时候我死盯着波浪线。
我把 `SpawnManager` 里限制屏幕对象的数值拨到了 500。那画面没法看。当我在满屏的弹片中引爆了一波重型碎片时，成百上千次 `Instantiate` 同时往内存里塞预制件，内存峰锯齿直接扎到顶，引爆了一次史诗级的 GC 处理。那一刻电脑的主线程被卡死了接近 20 毫秒，肉眼都能察觉到飞船愣了一下才接着跑。
要是在正儿八经的商业项目里，主程看到这波浪图能骂出声来。没办法，我加了个硬性约束去隐藏问题——同屏垃圾卡死在 80 个以下，就算性能再糟糕的笔记本目前也能咬住 60 帧不掉线。

## 6.2 交互边界防线
在测向心力算法时，我就想测试飞船能不能飞出世界地图。我一直死死按着外侧轨道扩张按键，但因为 $r_{target}$ 的边界常量在第一层数据验证里就被掐在了 15 个单位，它死活没能扯断数据外壳。这种纯数据的独裁压制，往往最能从根源上治好物理反常脱离。

# 第7章 总结与展望

## 7.1 跌跌撞撞的工程反思
从最开始不知道该怎么把 C# 的数学库变现成真实的推进力，到最后成功让一艘飞船平滑地在圆圈上滑行，这趟独自趟坑的开发确实让我受了点搓磨。我学到了很多在期末作业里没法体会的痛苦：框架设计比花里胡哨的特效重要一万倍。如果最开始没有强迫自己去配置 UniRx 消息总线，我保证这代码到中后期连我自己都不敢看了。每一块逻辑都在闭门造车，需要交流就发个声——这种剥离耦合的设计直接治好了开发层面上的一大部分后遗症。

## 7.2 后续不得不填的坑
我不能骗自己这项目算是写利索了。最刺眼的两个窟窿依然没来得及补。
第一个是**真的对象池（Object Pool）架构**。目前那些靠 `new` 和 `Destroy` 乱生乱死的太空垃圾一直在慢慢透支内存，要想突破同屏数量瓶颈，必须在一开局就压个几十兆的数组，用到谁激活谁，没用了抹去表现存着备用。没干这事只是因为我偷了点懒。
第二个是**更狂野的物理系统**。单向的吸引太公式化了。如果我懂计算网格或者会写底层的洛希极限断裂，当那些巨大的铁架子漂近黑洞时，它们就该在庞大的潮汐力拉扯下当场扭曲撕裂开来。要在单线程 C# 里手算这种级别的互相干涉显然不太现实，这恐怕得指望引擎哪天升级新的碰撞网格组件才能支撑得了这种规模演算。
